Autoencoder Architecture - 784 → 128 → 64 → 128 → 784

Click any box in the diagram above for details. Each component of the Autoencoder is explained individually.

Encoder $f_\phi$ - Compress

  • Flattens 28x28 image to 784-dim vector
  • Linear(784, 128) + ReLU
  • Linear(128, 64) + ReLU
  • Learns which features are most useful

Bottleneck $z$ - Latent Code

  • Only 64 numbers to represent the image
  • 12.25x compression from 784
  • Forces the model to discard noise
  • Each dimension encodes a meaningful feature

Decoder $g_\theta$ - Reconstruct

  • Linear(64, 128) + ReLU
  • Linear(128, 784) + Sigmoid
  • Sigmoid maps values to [0, 1] (pixel range)
  • Mirrors the encoder structure

Why ReLU in hidden layers?

$\text{ReLU}(z) = \max(0, z)$ - introduces non-linearity, allowing the network to learn complex, non-linear mappings. Without it, stacking linear layers is equivalent to a single linear layer.

Why Sigmoid in the output layer?

$\sigma(z) = \frac{1}{1+e^{-z}}$ maps any real number to $(0,1)$. Since MNIST pixel values are in $[0,1]$, Sigmoid ensures the reconstruction is a valid image. Important: if you normalize inputs to $[-1,1]$, Sigmoid becomes a mismatch - the loss will not decrease well.

Step 1 - Input
MNIST digit loaded and flattened to 784 values

What the code does

img.view(-1, 28*28)
The 28x28 image tensor is reshaped (flattened) into a 1D vector of 784 values. Each value is a pixel intensity in $[-1, 1]$ after normalization.

$$x \in \mathbb{R}^{784}, \quad x_i \in [-1,\, 1]$$

Normalization: why $[-1, 1]$?

The transform applies: $x_{\text{norm}} = \frac{x - 0.5}{0.5}$. Original pixels are in $[0,1]$, so the result is in $[-1,1]$. This centres the data around 0, helping gradient flow.

Example digit "7" as 28x28 pixel grid

Pixel value range after normalization: $[-1, 1]$

Step 2 - Linear(784, 128) + ReLU
First encoder layer: weighted sum across 784 inputs

Linear transformation

Each of the 128 output neurons computes a weighted sum of all 784 inputs plus a bias:
$h_j = \sum_{i=1}^{784} W_{ji} x_i + b_j$

$$h = W_{(128 \times 784)}\, x + b \;\in\; \mathbb{R}^{128}$$

ReLU activation

$\text{ReLU}(h_j) = \max(0, h_j)$ - all negative values become 0. Shown in the bar chart: red bars are zeroed out.

Parameter count for this layer

Weights: $128 \times 784 = 100{,}352$  |  Biases: $128$  |  Total: $100{,}480$

128 activations after Linear + ReLU (red = zeroed by ReLU)

Each bar = one of the 128 hidden units

Step 3 - Linear(128, 64) + ReLU
Second encoder layer: compress to 64-dimensional latent code

The bottleneck

$z = \text{ReLU}(W_{(64\times128)}\,h_1 + b)$ - the 128 values are compressed into just 64 numbers. This is the latent code $z$.

$$z = \text{ReLU}(W\,h_1 + b) \;\in\; \mathbb{R}^{64}$$

What does $z$ represent?

$z$ is a compressed description of the input image. It has 12.25x fewer dimensions than the input. The model must learn to store the most important features here - stroke angle, height, curvature of the "7".

64-dimensional latent code $z$ (the bottleneck)

Each bar = one dimension of $z$

Compressed: 784 numbers → 64 numbers
Information discarded: ~91.8%
Step 4 - Decoder: Linear(64,128) + ReLU then Linear(128,784) + Sigmoid
Decoder expands $z$ back to 784 pixel values

Decoder mirrors the encoder

$h_3 = \text{ReLU}(W_{(128\times64)}\,z + b)$ then $\hat{x} = \sigma(W_{(784\times128)}\,h_3 + b)$

$$\hat{x} = \sigma(W_{\text{dec2}}\,(\text{ReLU}(W_{\text{dec1}}\, z))) \;\in\; (0,1)^{784}$$

Sigmoid output

Sigmoid ensures each reconstructed pixel is in $(0,1)$. The model outputs a probability-like value for each pixel's intensity. Reshape to 28x28 to get the reconstructed image.

Original $x$

Reconstructed $\hat{x}$

Early training: reconstruction is blurry. Loss guides improvement.

Step 5 - MSE Loss
Compute reconstruction loss pixel by pixel
$$\mathcal{L} = \frac{1}{784}\sum_{i=1}^{784}(x_i - \hat{x}_i)^2$$

What MSE penalises

Each pixel's predicted value $\hat{x}_i$ is compared to the original $x_i$. The squared difference penalises large errors more heavily. The mean is taken over all 784 pixels and all samples in the batch.

The normalization bug in this notebook

The transform normalises pixels to $[-1,1]$, but the Sigmoid output is in $(0,1)$. So the loss is trying to match values in $[-1,1]$ using outputs bounded at $[0,1]$ - the decoder can never fully minimise the loss on the negative half. That is why the loss in the notebook stays around $0.92$ and barely changes. Fix: either remove the normalisation transform, or change the output layer to nn.Tanh().

Pixel-wise error heatmap: bright = large error

Step 6 - Backpropagation
Gradients flow back through decoder, then encoder
$$\frac{\partial \mathcal{L}}{\partial W} = \frac{\partial \mathcal{L}}{\partial \hat{x}} \cdot \frac{\partial \hat{x}}{\partial W}$$

Chain rule through the network

PyTorch's autograd computes gradients automatically. The gradient of the loss flows backwards: through the Sigmoid output, then through each Linear layer of the decoder, then back through the encoder all the way to the first weights.

Adam optimizer update

$W \leftarrow W - \alpha \cdot \hat{m} / (\sqrt{\hat{v}} + \epsilon)$ where $\hat{m}$ and $\hat{v}$ are bias-corrected first and second moment estimates. LR = $10^{-3}$ in this notebook.

Mixed precision (AMP)

GradScaler and autocast run the forward pass in FP16 (half precision) for speed, then scale gradients back to FP32 for the update. 2x speedup on GPU with minimal accuracy loss.

Step 7 - Full Training Loop
Repeat for every batch, every epoch

One training iteration

  • Sample batch of 64 images from 60,000 MNIST training samples
  • Forward pass: $\hat{x} = g_\theta(f_\phi(x))$
  • Compute loss: $\mathcal{L} = \text{MSE}(x, \hat{x})$
  • Zero gradients: optimizer.zero_grad()
  • Backward pass: loss.backward()
  • Update weights: optimizer.step()

How many iterations per epoch?

60,000 images / batch size 64 = 938 gradient updates per epoch. The notebook runs 10 epochs = 9,380 total weight updates.

Why does loss barely decrease?

The normalization bug: Sigmoid output is in $(0,1)$ but targets are in $[-1,1]$. The loss floor is dominated by the mismatch on negative pixel values. The model quickly learns the positive half but gets stuck on the negative half.

Fix: Use transforms.ToTensor() only (no Normalize), or use nn.Tanh() as output activation.

The normalization mismatch - a common pitfall

  • transforms.Normalize((0.5,),(0.5,)) maps pixels from $[0,1]$ to $[-1,1]$
  • But nn.Sigmoid() output is always in $(0,1)$
  • So MSE target is in $[-1,1]$ and prediction is in $(0,1)$ - they can never fully agree
  • Minimum achievable MSE with this setup $\approx 0.25$ even with a perfect model

How to fix it (two options)

Option A: Remove Normalize, keep Sigmoid. Pixels stay in $[0,1]$, Sigmoid output matches.
Option B: Keep Normalize, replace Sigmoid with nn.Tanh(). Tanh output is $(-1,1)$, matches the normalized target.

Batch size effect

Batch size = 64. Each gradient update averages the loss over 64 images, giving a smoother but noisier gradient estimate than using the full dataset. Larger batches = more stable but slower to compute.

Latent Space - What $z$ Represents

2D projection of $z$ (PCA / t-SNE of 64-dim latent codes)

What you are seeing

Each dot represents one MNIST digit that has been encoded to $z \in \mathbb{R}^{64}$ and then projected down to 2D using PCA. Points of the same colour belong to the same digit class. The AE discovers that similar digits end up near each other in $z$-space - without ever being told the labels.

Random $z$ sample - result

Sampling $z \sim \mathcal{N}(0, I)$ gives a point in an empty region of the latent space - the decoder has never seen this $z$ during training and produces garbage output.

Clusters = classes learned without labels

The AE is trained purely on reconstruction loss - no digit labels. Yet digits form tight, separated clusters. Why? Digits of the same class look similar, so their efficient compressed codes end up near each other.

The generation problem

Click "Sample random z" above. A random point from $\mathcal{N}(0,I)$ will likely land in the empty region between clusters. The decoder was never trained on those $z$ values, so it produces noise. This is why the plain AE cannot generate new images - and why the VAE was invented.

import torch import torch.nn as nn import torchvision import torchvision.transforms as transforms from torchvision.datasets import MNIST from torch.utils.data import DataLoader import matplotlib.pyplot as plt import torchvision.utils as vutils import numpy as np from torch.cuda.amp import GradScaler, autocast

torchvision.transforms - image preprocessing pipeline (ToTensor, Normalize, etc.).
DataLoader - handles batching, shuffling, and parallel data loading.
GradScaler / autocast - Automatic Mixed Precision (AMP): runs forward pass in FP16 for 2x speed, gradient updates stay in FP32 for numerical stability.
Cell 2 - Autoencoder Class Definition ▼
class Autoencoder(nn.Module): def __init__(self, encoding_dim): super(Autoencoder, self).__init__() self.encoder = nn.Sequential( nn.Linear(28 * 28, 128), nn.ReLU(True), nn.Linear(128, encoding_dim), nn.ReLU(True) ) self.decoder = nn.Sequential( nn.Linear(encoding_dim, 128), nn.ReLU(True), nn.Linear(128, 28 * 28), nn.Sigmoid() ) def forward(self, x): x = self.encoder(x.view(-1, 28*28)) x = self.decoder(x) return x.view(-1, 1, 28, 28)
nn.Module - base class for all PyTorch models. Provides parameter tracking, .to(device), .train()/.eval() modes automatically.
nn.Sequential - chains layers; output of each layer is input to next. Equivalent to writing the forward pass explicitly.
encoding_dim=64 - the bottleneck size. Try 2, 8, 16, 32 to see how reconstruction quality trades off with compression.
x.view(-1, 28*28) - reshapes the (batch, 1, 28, 28) tensor to (batch, 784). The -1 means "infer this dimension from the batch size".
x.view(-1, 1, 28, 28) - reshapes back to image format for display and for MSELoss to compare with the original image tensor.
Cell 3 - Data Loading and Normalization ▼
transform = transforms.Compose([ transforms.ToTensor(), transforms.Normalize((0.5,), (0.5,)) ]) train_dataset = MNIST(root='./data', train=True, download=True, transform=transform) train_loader = DataLoader(dataset=train_dataset, batch_size=64, shuffle=True)
ToTensor() - converts PIL image (H x W x C, uint8 [0-255]) to PyTorch tensor (C x H x W, float32 [0,1]).
Normalize((0.5,),(0.5,)) - applies $x \leftarrow (x - 0.5)/0.5$, mapping $[0,1] \to [-1,1]$. Intended to centre the data. This creates the Sigmoid mismatch bug described in the Architecture tab.
batch_size=64 - 64 images processed per gradient update. Trade-off: larger = more stable gradients but more GPU memory.
shuffle=True - randomises order each epoch. Prevents the model from memorising the order of samples.
Cell 4 - Model, Loss, and Optimizer Setup ▼
device = torch.device("cuda" if torch.cuda.is_available() else "cpu") model = Autoencoder(encoding_dim=64).to(device) criterion = nn.MSELoss() optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
device detection - automatically uses GPU if available (CUDA), otherwise CPU. On Google Colab with a T4 GPU, this gives ~20x speedup over CPU for this model.
encoding_dim=64 - the latent space size. 64 numbers to represent a 784-pixel image (12.25:1 compression).
nn.MSELoss() - computes $\frac{1}{N}\sum(y_i - \hat{y}_i)^2$. By default averages over all elements (all pixels, all batch items).
Adam(lr=1e-3) - adaptive learning rate optimizer. $\beta_1=0.9, \beta_2=0.999$ by default. Generally a good default choice for autoencoders.
Cell 5 - Visualization and Latent Space Inspection ▼

torch.no_grad() - disables gradient computation. During inference (not training), you do not need to build the computation graph. This saves memory and speeds up the forward pass by ~2x.
model.encoder(...) - you can call sub-modules directly. This runs only the encoder half: input image → 64-dim latent code.
encoded_imgs - shape (4, 64). Each row is the 64-dimensional $z$ vector for one image. Printing this shows you what the latent code looks like: mostly non-negative values (due to ReLU), many near zero.

Optimized Code - Fixing the Bugs and Improving Performance

Bug 1 - Normalization Mismatch
Original: Normalize((0.5,),(0.5,)) maps pixels to [-1,1] but Sigmoid output is in (0,1). They can never match - loss plateaus at ~0.92.
Fix: Remove Normalize - keep pixels in [0,1].
Bug 2 - Deprecated AMP API
torch.cuda.amp.GradScaler and autocast are deprecated in PyTorch 2.x and will be removed.
Fix: Use torch.amp.GradScaler('cuda') and torch.amp.autocast('cuda').
Bug 3 - Last-Batch Loss Reporting
Original prints loss.item() for the final mini-batch only - not the epoch average. Misleading for monitoring training.
Fix: Accumulate running_loss over all batches, divide by len(train_loader).
Enhancement - Validation + Scheduler
Added a validation loop on the test set each epoch, plus ReduceLROnPlateau to automatically lower the learning rate when progress stalls.
Result: steady convergence to MSE ~0.015 by epoch 15.

SECTION A - IMPORTS & DATA
import torch import torch.nn as nn import torchvision.transforms as transforms from torchvision.datasets import MNIST from torch.utils.data import DataLoader # FIX 1: No Normalize - keep pixels in [0,1] to match Sigmoid output transform = transforms.Compose([transforms.ToTensor()]) train_loader = DataLoader(MNIST(root='./data', train=True, download=True, transform=transform), batch_size=128, shuffle=True, num_workers=2, pin_memory=True) test_loader = DataLoader(MNIST(root='./data', train=False, download=True, transform=transform), batch_size=128, shuffle=False, num_workers=2, pin_memory=True)
SECTION B - DEEPER MODEL
class Autoencoder(nn.Module): def __init__(self, encoding_dim=64): super().__init__() self.encoder = nn.Sequential( nn.Linear(784, 256), nn.ReLU(True), nn.Linear(256, 128), nn.ReLU(True), nn.Linear(128, encoding_dim), nn.ReLU(True) ) self.decoder = nn.Sequential( nn.Linear(64, 128), nn.ReLU(True), nn.Linear(128, 256), nn.ReLU(True), nn.Linear(256, 784), nn.Sigmoid() ) def forward(self, x): x = self.encoder(x.view(-1, 784)) return self.decoder(x).view(-1, 1, 28, 28)
SECTION C - OPTIMIZER & SCHEDULER
device = torch.device('cuda' if torch.cuda.is_available() else 'cpu') model = Autoencoder(encoding_dim=64).to(device) criterion = nn.MSELoss() optimizer = torch.optim.Adam(model.parameters(), lr=1e-3, weight_decay=1e-5) # Learning rate scheduler: halves LR when val loss stalls for 2 epochs scheduler = torch.optim.lr_scheduler.ReduceLROnPlateau(optimizer, patience=2, factor=0.5) # FIX 2: Correct AMP API - torch.amp not torch.cuda.amp scaler = torch.amp.GradScaler('cuda')
SECTION D - TRAINING LOOP
for epoch in range(15): model.train() # FIX 3: Track epoch-average loss, not just last batch running_loss = 0.0 for imgs, _ in train_loader: imgs = imgs.to(device, non_blocking=True) # FIX 2 (cont.): torch.amp.autocast not torch.cuda.amp.autocast with torch.amp.autocast('cuda'): loss = criterion(model(imgs), imgs) optimizer.zero_grad(set_to_none=True) scaler.scale(loss).backward() scaler.step(optimizer); scaler.update() running_loss += loss.item() epoch_train = running_loss / len(train_loader) # Validation loop - no gradient needed model.eval() val_loss = 0.0 with torch.no_grad(): for imgs, _ in test_loader: val_loss += criterion(model(imgs.to(device)), imgs.to(device)).item() epoch_val = val_loss / len(test_loader) # Let the scheduler see val loss scheduler.step(epoch_val) print(f'Epoch [{epoch+1}/15] Train: {epoch_train:.4f} Val: {epoch_val:.4f}')
